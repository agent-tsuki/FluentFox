# File Structure

This document is the authoritative guide to the repository layout. It explains the purpose of every directory, the rules that govern it, and exactly where to put new code.

---

## Top-level Layout

```
FluentFox/
├── cmd/                  # Executable entry points (one binary per sub-directory)
├── internal/             # Private application code — 10 domains
├── pkg/                  # Reusable packages with no domain logic
├── db/                   # SQL migrations and seed data
├── content/              # MDX grammar content files (source of truth for chapters)
├── config/               # Typed environment config loader
├── docker/               # Dockerfile.dev and Dockerfile.prod
├── .github/workflows/    # CI pipelines
├── docs/                 # This documentation
├── .air.toml             # Hot-reload config (air)
├── .golangci.yml         # Linter 
├── .env.development      # Dev environment variables (never commit secrets)
├── docker-compose.yml    # Dev stack (Postgres + API)
├── docker-compose.test.yml  # Ephemeral test database
├── Makefile              # Single interface for all dev commands
├── go.mod / go.sum       # Module declaration and locked dependencies
└── railway.json          # Railway deployment config
```

---

## `cmd/` — Entry Points

Each sub-directory compiles into a separate binary. No business logic lives here — only wiring.

```
cmd/
├── api/
│   └── main.go      # HTTP server: loads config, wires all dependencies, starts router
└── sync-content/
    └── main.go      # CLI tool: parses MDX files and upserts chapters into the database
```

**Rules:**
- `cmd/api/main.go` is the only place where dependencies are instantiated and wired together.
- It must not contain any business logic, SQL, or HTTP handler code.
- If you add a new domain, register its routes here and nowhere else.

**Adding a new binary:** Create `cmd/<name>/main.go`. Keep it under ~100 lines.

## `internal/` — Domain Packages

This is where all application logic lives. The `internal/` restriction means nothing outside this module can import these packages.

Each domain follows an identical four-file structure:

```
internal/<domain>/
├── handler.go       # HTTP layer — parse request, call service, write response
├── service.go       # Business logic — orchestrates repos and external services
├── repository.go    # Data access — all SQL queries for this domain
├── model.go         # Types — DB models, request DTOs, response DTOs (three distinct sets)
└── test/
    └── handler_test.go  # Integration tests for this domain
```

### The Three Layers

| File | Imports | Never imports |
|------|---------|---------------|
| `handler.go` | `service`, `response`, `validator`, `middleware` | `pgx`, any DB driver |
| `service.go` | `repository`, `pkg/*`, other domain services | `net/http`, chi |
| `repository.go` | `pgx/v5`, `database/sql` | `net/http`, business types |

### The Three Model Sets (in `model.go`)

Every domain's `model.go` must keep these strictly separate:

```go
// 1. DB model — maps exactly to table columns, never sent to clients
type User struct {
    ID           uuid.UUID `db:"id"`
    PasswordHash string    `db:"password_hash"`
}

// 2. Request DTO — bound from JSON body, validated with struct tags
type RegisterRequest struct {
    Email    string `json:"email"    validate:"required,email"`
    Password string `json:"password" validate:"required,min=8"`
}

// 3. Response DTO — sent to clients, never contains passwords or internal fields
type RegisterResponse struct {
    ID    uuid.UUID `json:"id"`
    Email string    `json:"email"`
}
```

### Current Domains

| Domain | Responsibility |
|--------|---------------|
| `auth` | Registration, login, JWT tokens, password reset, email verification |
| `user` | Profile, settings, password change |
| `chapter` | Grammar chapters and concepts (read from DB, written by sync-content) |
| `srs` | Spaced repetition scheduling via go-fsrs — due cards, review submission |
| `quiz` | Quiz sessions, answer submission, session completion |
| `progress` | Chapter and vocabulary progress tracking |
| `streak` | Daily streak tracking and freeze logic |
| `xp` | XP transactions, levelling, leaderboard |
| `shop` | In-app shop items, purchases, user inventory |
| `admin` | Admin dashboard, moderation, RBAC-protected routes |

### Adding a New Domain

1. Create `internal/<domain>/` with all four files.
2. Wire it in `cmd/api/main.go`: instantiate repo → service → handler, register routes.
3. Add migrations in `db/migrations/` if new tables are needed.
4. Write tests in `internal/<domain>/test/handler_test.go`.

### `internal/middleware/`

Middleware is shared infrastructure, not a domain. It only reads from the request — it never writes to the database.

```
internal/middleware/
├── auth.go       # Validates JWT, injects claims into context
├── rbac.go       # Role check — requires admin or superadmin
├── cors.go       # CORS headers, reads AllowedOrigins from config
├── logger.go     # Request/response logging via zap
└── ratelimit.go  # Token bucket (10 req/min default)
```

### `internal/mdxparser/`

Used only by `cmd/sync-content`. Parses MDX files into Go structs that are then upserted into the database. This package has no HTTP dependency.

## `pkg/` — Shared Packages

These packages are reusable within the module. They contain no domain logic — they are infrastructure primitives.

```
pkg/
├── database/
│   ├── postgres.go      # pgxpool setup with connection limits and health checks
│   └── transaction.go   # Helper for running multiple queries in one transaction
├── token/
│   ├── jwt.go           # JWT generation and validation (access tokens)
│   └── hash.go          # SHA-256 hashing (refresh tokens, email tokens)
├── mailer/
│   ├── interface.go     # Mailer interface — swap implementations without touching callers
│   └── resend.go        # Resend API implementation
├── storage/
│   ├── interface.go     # Storage interface
│   └── r2.go            # Cloudflare R2 implementation (S3-compatible)
├── validator/
│   └── validator.go     # go-playground/validator wrapper with custom error messages
├── cache/
│   ├── interface.go     # Cache interface
│   └── memory.go        # In-memory cache (chapter content loaded at startup)
└── response/
    └── response.go      # Standard JSON envelope helpers
```

**Rules:**
- `pkg/` packages must not import `internal/` packages — only the other way around.
- Keep interfaces in `interface.go`. If you add a second implementation (e.g., Redis cache), add `redis.go` alongside `memory.go`.

### `pkg/response` — API Envelope

All responses go through this package. Never write `json.Encode` directly in handlers.

```go
// Success
response.JSON(w, http.StatusOK, RegisterResponse{...})

// Success with pagination
response.JSONWithMeta(w, http.StatusOK, items, meta)

// Errors
response.BadRequest(w, "invalid_input", "Email is required")
response.Unauthorized(w)
response.NotFound(w)
response.InternalServerError(w)
```

Wire format:
```json
{ "data": { ... } }
{ "data": [...], "meta": { "page": 1, "total": 42 } }
{ "error": { "code": "invalid_input", "message": "Email is required" } }
```

## `db/` — Database

```
db/
├── migrations/          # SQL migration files (golang-migrate format)
│   ├── 000001_create_enum_types.up.sql
│   ├── 000001_create_enum_types.down.sql
│   └── ...              # 72 files total (36 pairs)
└── seeds/               # Idempotent seed data (safe to run multiple times)
    ├── characters.sql   # Hiragana and Katakana
    ├── kanji.sql        # Kanji with readings and meanings
    └── xp_config.sql    # XP reward amounts by source type
```

**Migration naming:** `<sequence>_<description>.{up,down}.sql`

```bash
make migrate-new name=add_user_avatar   # creates 000037_add_user_avatar.{up,down}.sql
```

**Rules:**
- Every `up` migration must have a corresponding `down` migration.
- Never edit a migration that has already been applied to production. Create a new one.
- Seed files must use `INSERT ... ON CONFLICT DO NOTHING` or equivalent — they run in CI.

## `content/` — Grammar Content

```
content/
└── grammar/
    ├── N5/
    │   ├── chapter-01.mdx
    │   └── chapter-02.mdx
    └── N4/
        └── chapter-01.mdx
```

These are the **source of truth** for chapter content. They are not served directly — `cmd/sync-content` parses them and upserts into the database. Editors modify MDX files; the API reads from the database.

---

## `config/`

```
config/
└── config.go    # Typed Config struct + Load() function
```

`config.Load()` is called once at startup in `cmd/api/main.go`. It panics on any missing required variable so the app never starts with a broken configuration. No package calls `os.Getenv` directly — everything flows through `Config`.

## Where to Put New Code — Quick Reference

| What you're adding | Where it goes |
|---|---|
| New HTTP endpoint in an existing domain | Add method to `internal/<domain>/handler.go`, register route in `cmd/api/main.go` |
| New business rule | `internal/<domain>/service.go` |
| New SQL query | `internal/<domain>/repository.go` |
| New database table | `db/migrations/` (new pair), update model in `internal/<domain>/model.go` |
| New shared utility (no domain logic) | `pkg/<name>/` |
| New external service integration | `pkg/<name>/interface.go` + implementation file |
| New environment variable | `config/config.go` Config struct + `Load()`, then `.env.development` |
| New binary / CLI tool | `cmd/<name>/main.go` |
| New domain from scratch | `internal/<domain>/` (4 files) + wire in `cmd/api/main.go` |
| New MDX grammar content | `content/grammar/<level>/chapter-XX.mdx`, then run `make sync-content` |
| New seed data | `db/seeds/<name>.sql` (must be idempotent) |
