# Request & Data Flow (Development)

This document shows how data moves through the system. All diagrams reflect the **local development** setup — Docker Compose + hot reload. Production routing (Railway, TLS, etc.) is not covered here.

---

## Development Environment Topology

```mermaid
graph TD
    Dev["Developer Machine<br/>(browser / curl / Postman)"]
    API["API Container<br/>localhost:8080<br/>(air hot-reload)"]
    PG["Postgres Container<br/>localhost:5432<br/>fluentfox_dev"]
    ENV[".env.development"]
    MDX["content/grammar/**/*.mdx<br/>(mounted volume)"]

    Dev -->|"HTTP request"| API
    API -->|"pgxpool queries"| PG
    ENV -->|"env vars at startup"| API
    MDX -->|"make sync-content<br/>(one-shot CLI)"| PG
```

Both containers are started with `make dev` (`docker compose up`). Source code is bind-mounted into the API container, so `air` detects file changes and rebuilds without restarting Docker.

## Startup Sequence

```mermaid
sequenceDiagram
    participant OS as Host OS
    participant Main as cmd/api/main.go
    participant Cfg as config.Load()
    participant DB as PostgreSQL
    participant Cache as pkg/cache
    participant Router as chi.Router

    OS->>Main: docker compose up → air → ./tmp/server
    Main->>Cfg: Load() — reads .env.development
    Cfg-->>Main: *Config (panics if any var missing)
    Main->>DB: pgxpool.New() + Ping()
    DB-->>Main: connection pool ready
    Main->>Cache: WarmUp() — loads all chapters into memory
    DB-->>Cache: chapter rows
    Main->>Router: register all middleware + routes
    Router-->>Main: http.Server ready
    Main->>OS: listening on :8080
```

If `config.Load()` panics, the container exits immediately — check `.env.development` for missing variables.

## Authenticated Request Flow

This is the path for any route that requires a logged-in user (e.g., `GET /srs/due`).

```mermaid
sequenceDiagram
    participant Client as Client
    participant MW as Middleware Stack
    participant H as Handler
    participant S as Service
    participant R as Repository
    participant DB as PostgreSQL

    Client->>MW: GET /srs/due<br/>Authorization: Bearer <access_token>

    Note over MW: 1. Logger — logs method, path, IP
    Note over MW: 2. RateLimiter — 10 req/min per IP
    Note over MW: 3. CORS — sets response headers
    Note over MW: 4. RequireAuth — validates JWT

    MW->>MW: parse JWT → extract user_id, role, email_verified
    alt JWT invalid or expired
        MW-->>Client: 401 Unauthorized
    end
    MW->>H: r.Context() now contains user claims

    H->>H: middleware.ContextUserID(ctx) → userID
    H->>S: svc.GetDueCards(ctx, userID, limit)
    S->>R: repo.GetDueCards(ctx, userID, now, limit)
    R->>DB: SELECT ... FROM srs_cards WHERE user_id=$1 AND due <= $2
    DB-->>R: []Card rows
    R-->>S: []Card
    S->>S: filter, sort, apply business rules
    S-->>H: []DueCardResponse
    H->>Client: 200 {"data": [...]}
```

## Authentication Flow — Register

```mermaid
sequenceDiagram
    participant Client as Client
    participant RL as RateLimiter
    participant H as auth.Handler
    participant S as auth.Service
    participant UR as user.Repository
    participant AR as auth.Repository
    participant Mail as pkg/mailer (Resend)
    participant DB as PostgreSQL

    Client->>RL: POST /auth/register {"email","password","username"}
    RL->>H: (passes through — within limit)
    H->>H: decode JSON + validate struct tags
    alt validation fails
        H-->>Client: 400 {"error":{"code":"validation_error",...}}
    end

    H->>S: svc.Register(ctx, req)
    S->>UR: userRepo.GetByEmail(ctx, email)
    alt email already exists
        S-->>H: ErrEmailAlreadyExists
        H-->>Client: 409 Conflict
    end

    S->>S: bcrypt.GenerateFromPassword(password, 12)
    S->>UR: userRepo.Create(ctx, user) → DB INSERT
    S->>AR: authRepo.CreateEmailVerification(ctx, userID, tokenHash, expiresAt)
    S->>Mail: mailer.SendVerificationEmail(email, token)
    Mail-->>S: sent (async — does not block response)
    S-->>H: RegisterResponse{id, email}
    H-->>Client: 201 {"data":{"id":"...","email":"..."}}
```

## Authentication Flow — Login & Token Refresh

```mermaid
sequenceDiagram
    participant Client as Client
    participant H as auth.Handler
    participant S as auth.Service
    participant R as auth.Repository
    participant DB as PostgreSQL

    Client->>H: POST /auth/login {"email","password"}
    H->>S: svc.Login(ctx, req)
    S->>R: repo.GetUserByEmail(ctx, email)
    alt user not found
        S-->>H: ErrInvalidCredentials
        H-->>Client: 401 (generic — do not reveal which field is wrong)
    end
    S->>S: bcrypt.CompareHashAndPassword()
    alt wrong password
        S-->>H: ErrInvalidCredentials
        H-->>Client: 401
    end

    S->>S: token.GenerateAccessToken(userID, role) → JWT (60min)
    S->>S: token.GenerateRefreshToken() → random UUID
    S->>S: hash(refreshToken) → store hash only
    S->>R: repo.CreateRefreshToken(ctx, userID, hash, expiresAt)
    S-->>H: LoginResponse{access_token, refresh_token}
    H-->>Client: 200 {"data":{"access_token":"eyJ...","refresh_token":"uuid"}}

    Note over Client: Later — access token expires

    Client->>H: POST /auth/refresh {"refresh_token":"uuid"}
    H->>S: svc.RefreshToken(ctx, req)
    S->>S: hash(refreshToken)
    S->>R: repo.GetRefreshToken(ctx, hash)
    alt token not found or revoked or expired
        S-->>H: ErrInvalidToken
        H-->>Client: 401
    end
    S->>R: repo.RevokeRefreshToken(ctx, oldTokenID)
    S->>S: generate new access + refresh token pair (rotation)
    S->>R: repo.CreateRefreshToken(ctx, ...)
    S-->>H: new token pair
    H-->>Client: 200 {"data":{...new tokens...}}
```

## Content Sync Flow (MDX → Database)

This is a one-shot CLI command, not an HTTP request. Run it whenever MDX files change.

```mermaid
sequenceDiagram
    participant Dev as Developer
    participant CLI as cmd/sync-content
    participant Parser as internal/mdxparser
    participant DB as PostgreSQL

    Dev->>CLI: make sync-content
    CLI->>CLI: config.Load() + pgxpool.New()
    CLI->>Parser: parser.ParseDirectory("content/grammar")

    loop for each .mdx file
        Parser->>Parser: read file → extract frontmatter (slug, title, jlpt_level, order)
        Parser->>Parser: parse concepts, vocabulary, examples, segments, sidebar
        Parser-->>CLI: Chapter struct
    end

    loop for each Chapter
        CLI->>DB: INSERT INTO chapters ... ON CONFLICT (slug) DO UPDATE
        CLI->>DB: INSERT INTO concepts ... ON CONFLICT DO UPDATE
        CLI->>DB: INSERT INTO vocabulary ... ON CONFLICT DO UPDATE
        CLI->>DB: INSERT INTO examples ... ON CONFLICT DO UPDATE
    end

    CLI-->>Dev: sync complete (idempotent — safe to run repeatedly)
```

## Admin Request Flow

Admin routes require both a valid JWT **and** the role `admin` or `superadmin`. Two middleware layers enforce this.

```mermaid
sequenceDiagram
    participant Client as Admin Client
    participant Auth as RequireAuth MW
    participant RBAC as RequireRole MW
    participant H as admin.Handler
    participant S as admin.Service

    Client->>Auth: GET /admin/users<br/>Authorization: Bearer <token>
    Auth->>Auth: validate JWT → user_id, role="editor"
    Auth->>RBAC: pass to RequireRole("admin","superadmin")
    RBAC->>RBAC: check role — "editor" not in allowed set
    RBAC-->>Client: 403 Forbidden

    Note over Client: With role="admin"

    Client->>Auth: GET /admin/users<br/>Authorization: Bearer <admin_token>
    Auth->>RBAC: role="admin" ✓
    RBAC->>H: handler receives request
    H->>S: svc.ListUsers(ctx, filters)
    S-->>H: []UserAdminView
    H-->>Client: 200 {"data":[...]}
```

## Data Layer — pgxpool Connection Lifecycle

```mermaid
graph LR
    Start["App startup"] --> Pool["pgxpool.New()<br/>min=5, max=25 connections"]
    Pool --> HC["Health check every 60s"]
    Pool --> Q["Query request"]
    Q --> Acq["Acquire connection from pool"]
    Acq --> Exec["Execute SQL"]
    Exec --> Rel["Release back to pool"]
    HC -->|"connection dead"| Evict["Evict + reconnect"]
    Pool -->|"idle > 30min"| Close["Close idle connection"]
    Pool -->|"open > 1h"| Rotate["Rotate connection"]
```

Connection limits are set via env vars (`DB_MAX_CONNS=25`, `DB_MIN_CONNS=5`). Adjust these based on your Postgres plan's connection limit.
